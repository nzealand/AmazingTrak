## INVARIANTS

**Adding a site setting touches exactly five places.** Miss one and it fails silently or panics at scan time:
1. `db.go` ŌĆö additive `ALTER TABLE site_preferences ADD COLUMN x INTEGER NOT NULL DEFAULT 0` in the `applySchema` migration block (bare `db.Exec`, errors ignored, idempotent by design).
2. `models.go` ŌĆö field on the `SitePreferences` struct.
3. `models.go` `getSitePrefs` ŌĆö add to **both** the `COALESCE(...)` SELECT list and the positional `Scan(...)` args. These are order-dependent and unchecked by the compiler; a mismatch is a runtime error only.
4. `handlers_admin.go` ŌĆö a `case "<action>":` in the `handleAdminSettingsPost` switch (~line 1858). Unknown actions fall through to `default: setFlash("Unknown action.")` ŌĆö a missing case looks like a working form that silently does nothing.
5. `templates/admin/settings.html` ŌĆö a `<section>` with `<input type="hidden" name="action" value="<action>">`.

**Boolean setting form idiom** (`templates/admin/settings.html`, `handlers_admin.go`): a hidden `<input value="0">` immediately followed by `<input type="checkbox" value="1">` with the same `name`, read server-side via `lastFormValue(r, "name") == "1"` and stored with `boolToInt(...)`. `lastFormValue` (not `r.FormValue`) is what makes the unchecked case work ŌĆö `FormValue` returns the *first* value, which is always "0".

**Admin version string is hardcoded in a template, not Go.** `templates/admin_base.html` line 13, inside the `.brand` span (`v3.12.2 Jun 26 12:00PM`). Per CLAUDE.md: refresh the *timestamp* after any significant change; only *increment the number* (and add a `===3.x.y=====` CHANGELOG block) when the user explicitly asks.

**Inline `<script>` needs `nonce="{{.Nonce}}"`** (CSP is enforced). Scripts loaded from a CDN `src` do not. See `templates/map.html`.

**No tests, no linter.** Verification means: `go build -o /tmp/.../att .`, `sqlite3 amazingtrak.db ".backup '/tmp/.../test.db'"`, then run the binary with `PORT=8099 DB_PATH=<copy> UPLOADS_DIR=<tmp>` and curl it. Never point a test run at `amazingtrak.db` ŌĆö it's the live-ish DB and WAL files sit beside it.

**App binds `127.0.0.1` by default** (`BIND_ADDR` overrides), so it's only reachable via nginx in prod; local curl must use `127.0.0.1`, not the LAN IP.

## GOTCHAS

**`trains.train_number` is unique *per corridor*, not globally.** The UNIQUE constraint is `(corridor_id, train_number)`. ~590 distinct numbers exist and they collide across corridors, so any external feed keyed on bare train number needs a second disambiguator (route/corridor name) or it will attach data to the wrong corridor's page. This is the single biggest footgun for anything joining outside data to `trains`.

**`stops.latitude` / `stops.longitude` exist but are 100% NULL** in the real database (verified across all 44 corridors with stops). Nothing populates them. Route geometry on `/map` does **not** come from `stops` ŌĆö it comes from USDOT NTAD GeoJSON fetched at runtime. Don't assume the columns being present means they're usable; anything needing station coordinates must import them first (Amtrak static GTFS `stops.txt` at `content.amtrak.com/content/gtfs/GTFS.zip` is the obvious source).

**`publicPage.Data` is `interface{}`, and templates index into it directly.** `templates/map.html` did `{{range .Data}}` over a bare `[]Corridor`. Changing what a handler puts in `Data` silently breaks the template at render time, not compile time ŌĆö every consumer of `.Data` in that template must be updated together.

**`/api/amtrak-routes` has a 12-hour process-lifetime memory cache** (`amtrakRoutesJSON` / `amtrakRoutesMu` / `amtrakRoutesFetched` in `handlers_public.go`), populated lazily on first request, and it pages the ArcGIS FeatureServer 1000 features at a time. It is *not* re-fetched on restart until someone hits `/map`.

**`db.go`'s migration block ignores every error.** Statements are bare `db.Exec(...)` with no error check ŌĆö that's deliberate (re-running `ALTER TABLE ADD COLUMN` on an existing column errors harmlessly, which is what makes `applySchema` idempotent on the running prod instance). Consequence: a genuinely malformed migration also fails silently. Verify new columns actually landed.

**Comment/`comments` table was rebuilt via the create-new/copy/drop/rename dance** (`db.go` ~line 160) because SQLite can't ALTER a CHECK constraint. Note the related workaround: `suggestions.is_spam` is a separate flag column rather than a new `status` enum value, for exactly this reason. Follow that precedent ŌĆö layer a flag rather than extend a CHECK.

## ARCHITECTURE

**`site_preferences` is a single-row table (`WHERE id=1`)**, extended horizontally by `ALTER TABLE ADD COLUMN` over time rather than being a key/value store. Every runtime-toggleable behavior lives there (email on/off, rate limits, themes, auto-approval policy, and now live trains). `getSitePrefs(app.db)` is the only read path; it returns hardcoded defaults on `sql.ErrNoRows`.

**Feature toggles gate at the handler, not the router.** Routes are registered unconditionally in `main.go`; the handler checks the pref and 404s if disabled. Background pollers check the pref each tick and no-op rather than being conditionally started ŌĆö a setting change therefore takes effect without a restart, bounded by the poll interval.

**Admin nav is permission-gated in `templates/admin_base.html`** via `{{if or (eq .PermLevel 0) (ge .PermLevel N)}}`, with PermLevel 0 = central admin (all access). Settings and Email Errors are level 5; Users is 6; Routes/Conductors 4; Trains 3. `main.go` enforces the same numbers server-side with `app.requirePermission(N, handler)` ŌĆö the template check is cosmetic, the mux check is real. Both must be updated together.

**`/map` (`templates/map.html`) was already a full Leaflet map before this session**: Leaflet 1.9.4 from jsdelivr CDN, OpenStreetMap tiles with attribution, and one colored GeoJSON polyline per Amtrak route from USDOT. It carries a JS-side `CORRIDOR_BY_NORM` slug index that maps a feed's route name back to a local corridor slug by stripping an `amtrak-` prefix ŌĆö the same normalize-and-match problem exists on both the JS and Go sides now.

**Upstream Amtraker v3 feed shape** (`api-v3.amtraker.com/v3/trains`, consumed in `livetrains.go`): a JSON object keyed by train number, each value an **array** of currently-in-service runs of that number (a daily long-distance train has several simultaneously). `trainState` Ōłł {`Active`, `Predeparture`, `Completed`}; station `status` Ōłł {`Departed`, `Enroute`, `Station`}. Delay = the first non-`Departed` station's `arr` minus `schArr` (both RFC3339 with offset). The feed also carries Brightline (`b`-prefixed) and VIA Rail (`v`-prefixed) train numbers, which will never match this DB.
