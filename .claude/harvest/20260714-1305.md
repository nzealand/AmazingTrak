## INVARIANTS

**Prod deploy is `sudo ./recompile-deploy.sh` run *on the droplet*, never from the dev machine.** It: stops the `amazingtrak` systemd service → `PRAGMA wal_checkpoint(TRUNCATE)` on the live DB → full dir snapshot to `../amazingtrak-backup` → writes/refreshes tiered rollback bundles (`daily`/`weekly`/`monthly`/`yearly` under `/home/nv/backups/amazingtrak`, each a self-contained `amazingtrak.db` + binary + `.env` + `uploads/`, refreshed by *age* not calendar day) → restarts the service immediately (site back up on the old binary while building) → `git pull --ff-only` as the `nv` user → `go build -o amazingtrak ./...` as the `nv` user → `systemctl restart amazingtrak` → health-checks `/healthz` for 20s. A code change sitting in `origin/main` is not live until this runs — pushing alone does nothing.

**Dev restart is `./recompile-dev.sh`** (no sudo, no systemd): `fuser -k ${PORT}/tcp` to free the port, then `nohup go run . > app.log 2>&1 &`, then health-check loop. `--build` compiles only; `--run` runs in foreground. This is the only thing that makes local source edits visible — a `go run .` process started earlier does **not** pick up new edits by itself.

## GOTCHAS

**The dev machine (`/home/nv/amazingtrak`, port 8000, `DEV=true` in `.env`) and prod (`foamer.online`, DigitalOcean droplet) are two entirely separate running instances with separate SQLite databases.** Toggling a `site_preferences` flag, or verifying a page renders correctly, on dev proves nothing about prod's DB state. Don't infer prod behavior from a dev-machine test — check the live URL directly.

**A missing feature on a live page can mean "not deployed" or can mean "no data for this specific entity" — check the latter before assuming the former.** Concretely: `GET /trains/<slug>` shows no live-status badge whenever that train isn't in the current Amtraker snapshot (predeparture / not running today), which is indistinguishable at a glance from the new code simply not being deployed yet. Diagnose by hitting `/api/live-trains` directly and grepping for the train's slug — if it's absent from the JSON, the page is correct and there's nothing to fix; if it's present but the badge still doesn't render, *then* the binary is stale.

**Feature-toggle changes (`site_preferences.live_trains_enabled`) take effect without a restart, bounded by the 90s poll tick** — the background poller in `livetrains.go` re-checks the DB flag every tick rather than being started/stopped. Actual *code* changes (new template markup, new Go logic) still require the full rebuild-and-restart in recompile-deploy.sh; the two are independent axes and it's easy to conflate "the toggle is on" with "the new code is running."

**`delayStatus()` in `livetrains.go` buckets only on the *magnitude* of lateness, not sign** — `>30` min late → `verylate`, `>10` → `late`, everything else (including a train running 29 min *early*) → `ontime`/green. This was already true for map marker colors; it's now also true anywhere else that reuses `delayStatus`/the `Status` field (e.g. a per-train badge), so an early train and a slightly-late train render identically.

## ARCHITECTURE

**Corridor-level on-time performance and per-train live delay status are two unrelated data sources that happen to sound similar.** `corridors.on_time_percent` (`db.go`, `models.go`, `handlers_admin.go` ~223) is a static, admin-edited quarterly figure sourced from FRA performance reports — no job, no live component, just a number someone typed in. Per-train "X min late right now" comes entirely from the existing 90s Amtraker poll in `livetrains.go` (`DelayMin`/`Status`/`NextStation` on `liveTrain`) and requires no new schema or job — it was already being computed for map markers, just not displayed anywhere else. Adding it to the train detail page was a matter of scanning the in-memory snapshot by slug (`findLiveTrain`, linear scan over ~100-130 entries, called fresh on every page render — no caching, cheap at this scale) and reusing the existing bucket/format logic, not adding infrastructure.
