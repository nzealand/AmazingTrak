## INVARIANTS

- Data corrections must be shipped as code (a numbered migration and/or a `seed.go` fix), never as a direct edit to the local `amazingtrak.db` — the user doesn't use the local DB and the only deploy path is `foam.sh` (push to GitHub) → `drop.sh` + `./recompile` on the server (code only, no file copy). Editing the local SQLite file accomplishes nothing for production.
- The admin version string is a hand-edited literal in `templates/admin_base.html` (`v3.12.X <Mon> <D> <H:MM><AM/PM>`, e.g. `v3.12.4 Jul 14 4:18PM`) — not derived from a build const or git tag. Bump it manually after every significant change.
- Numbered migrations follow the pattern in `db.go`: `migrationApplied(db, N)` / `markMigration(db, N)`, backed by a `schema_migrations` table, added below the comment `── Versioned migrations (non-idempotent) ──`. Match by stable business key (e.g. `train_number`, corridor `slug`), never by row `id` — ids differ between any two copies of the DB (dev vs. production).

## GOTCHAS

- **Migration/seed ordering trap**: `openDB()` (`db.go:10`) runs `applySchema` then `runMigrations` and returns; `seedDB` is called *afterward*, in `main.go:680`, only on a fresh install. So a migration written to "correct bad seeded data" runs *before* the seed rows exist on a brand-new DB — it matches 0 rows, gets marked applied, and then the seed step inserts the bad data anyway, permanently uncorrected (the migration will never fire again since it's marked done). It only self-heals **existing/production** databases (already seeded from a previous, buggy version) because there the migration runs against real rows. Conclusion: when a migration exists to fix bad seeded data, the root cause in `seed.go` must also be fixed directly, or fresh installs stay broken forever.
- `pollLiveTrains` (`livetrains.go:311`) does one immediate `poll()` at process startup in addition to the 90s ticker, but that immediate poll reads `liveTrainsEnabled()` at the instant it fires. Flipping the DB flag on after the process is already running does nothing until the next 90s tick — for interactive testing you must restart the process after enabling the flag, not just wait a couple seconds.
- Killing a `go run .` dev server by `pkill -f "go run ."` is unreliable when multiple dev servers/ports are in play (e.g. one already running from a previous session on 8080) — it can miss the target or kill the wrong one. Use `lsof -t -i:<port>` to find and kill the exact process bound to the test port instead.
- `go run .` compiles to a temp binary under `/tmp/go-build.../exe/amazingtrak` and that process keeps running (and the port stays bound) even after the shell tool call that launched it "returns" if launched via a bare foreground Bash call that times out — always launch long-running dev servers with `run_in_background: true` and kill by port, not by assuming the shell call's exit ended the process.

## ARCHITECTURE

- Live train tracking (`livetrains.go`) pulls from the free community "Amtraker v3" API (`api-v3.amtraker.com/v3/trains`), gated entirely behind `site_preferences.live_trains_enabled` — while off, the code never calls out to the API at all (checked both in the poller and in `handleLiveTrains`). Data is cached in-memory (`liveTrainsCache`, not the DB) with a 10-minute max-age staleness cutoff; an upstream outage degrades to "no live data shown" rather than freezing stale positions on the map.
- Matching an upstream Amtraker train to a local DB train is by train number first; if a number is ambiguous across corridors (same number reused in two different named services upstream vs. locally), it disambiguates using a normalized route-name comparison plus a hardcoded `routeAliases` map (`livetrains.go:93`) for known upstream naming mismatches (typos like "Northest Regional", or upstream splitting one corridor into two names, e.g. Illini/Saluki). An unresolvable ambiguous match is dropped rather than guessed.
- Train number uniqueness (`UNIQUE(corridor_id, train_number)` in `trains` table) is per-corridor only, confirmed again via `matchTrain`'s disambiguation path — the same train number can legitimately appear in two different corridors' seed lists (this was the root cause of the Acela/Northeast Regional data bug: 816/817/880/806/807/887/888 and 2107/2117 were simply seeded under the wrong one of the two corridors in `seed.go`'s `trainSeeds` table).
- Corridor detail pages (`handlers_public.go: handleCorridor`) and the routes list (`handleCorridors`) build their own small per-request lookup maps (`liveByTrain map[string]*liveTrain`, `liveCounts map[string]int`) from the single cached snapshot rather than calling into per-train live lookups repeatedly — one pass over `app.liveTrains.load()` per page render.
